# True North Atlas — Deployment Manual

Step-by-step instructions from a zip file on your laptop to a live, publicly
accessible analytics platform and block explorer at tnorth.com, synced from
your Start9 node.

**Time required:** ~45 minutes of hands-on work, then the initial chain sync
runs unattended for several days.

---

## Part 0 — Prerequisites

Before starting, have these ready:

- [ ] A **GitHub account** and `git` installed locally (`git --version` to check)
- [ ] A **Render account** (render.com) with a payment method attached — the
      database and services in this deployment are paid-tier
- [ ] Your **Start9 server** running Bitcoin Core 25 or newer (you're on 29.3 ✓),
      fully synced
- [ ] Optional but recommended: a free **CryptoCompare API key**
      (min-api.cryptocompare.com) — raises rate limits for the price-history sync

### 0.1 — Verify your node's RPC settings

1. Open your Start9 dashboard → **Services → Bitcoin Core**.
2. Under **Config → RPC Settings**, confirm RPC is **enabled**.
3. Open **Properties** and copy down:
   - **RPC Username**
   - **RPC Password**
4. Open **Interfaces** and copy down:
   - The **LAN address** (used if you run the sync worker at home — Part 4,
     Option A, recommended)
   - The **Tor address** (`xxxx…xxxx.onion`, port 8332 — used for Option B)

> `txindex` is **not** required. The app fetches blocks with
> `getblock <hash> 3` and resolves transactions through its own UTXO index.

## Part 1 — Put the code on GitHub

1. Unzip `truenorth-atlas.zip`. You should see this structure:

   ```
   truenorth-atlas/
   ├── render.yaml          ← Render Blueprint (infrastructure as code)
   ├── README.md
   ├── DEPLOYMENT.md        ← this manual
   ├── .env.example         ← every environment variable, documented
   ├── .gitignore
   ├── server/              ← API + sync worker
   └── web/                 ← frontend
   ```

2. Create an **empty** repository on GitHub: github.com → **＋ → New
   repository** → name it `truenorth-atlas` → leave "Add a README" and
   ".gitignore" **unchecked** → **Create repository**.

3. In a terminal, from inside the unzipped folder:

   ```bash
   cd truenorth-atlas
   git init
   git add .
   git commit -m "True North Atlas: analytics + explorer + private API"
   git branch -M main
   git remote add origin https://github.com/YOUR_USERNAME/truenorth-atlas.git
   git push -u origin main
   ```

4. When prompted to authenticate, use a **personal access token** as the
   password (GitHub → Settings → Developer settings → Fine-grained tokens →
   generate one with *Contents: read/write* on this repo). GitHub no longer
   accepts account passwords for git. If you have the GitHub CLI: `gh auth
   login` once instead.

5. Refresh the GitHub page — you should see all 50+ files.

## Part 2 — Deploy the Blueprint on Render

1. Log in to render.com → **New → Blueprint**.
2. Click **Connect account** if this is your first time, authorize Render to
   see the `truenorth-atlas` repo, then **select the repo**.
3. Render parses `render.yaml` and shows a plan:

   | Resource       | Type              | What it is                       |
   |----------------|-------------------|----------------------------------|
   | `atlas-db`   | PostgreSQL 16     | 150 GB disk — holds the UTXO set |
   | `atlas-api`  | Web service       | The read API + explorer + admin  |
   | `atlas-sync` | Background worker | Replays the chain from your node |
   | `atlas-web`  | Static site       | The frontend                     |

4. Render will prompt for the env vars marked `sync: false`. **You can leave
   them blank for now** — Part 3 fills them in properly. Click **Apply**.
5. Wait for the database and services to provision (~5 minutes). `atlas-api`
   and `atlas-web` will go live; `atlas-sync` will crash-loop until it has
   node credentials — that's expected at this stage.

## Part 3 — Configure environment variables

Open each service's **Environment** tab in the Render dashboard.

### 3.1 — `atlas-api`

| Variable | Value |
|---|---|
| `ADMIN_TOKEN` | Auto-generated by the Blueprint — **copy it somewhere safe now**; it unlocks the admin panel (Part 6) |
| `CORS_ORIGIN` | Already set to `https://tnorth.com,https://www.tnorth.com` — add any other origins that will embed the app |
| `BITCOIN_RPC_URL` / `USER` / `PASSWORD` | *Optional.* Fill these (same values as the worker, Part 4) to enable full transaction detail in the explorer. Without them, the explorer serves local-index data and flags `"rpc": false` |

### 3.2 — `atlas-web`

| Variable | Value |
|---|---|
| `VITE_API_URL` | The `atlas-api` URL, e.g. `https://atlas-api.onrender.com` (find it at the top of the service page) |

After setting it, click **Manual Deploy → Deploy latest commit** on
`atlas-web` — the URL is baked in at build time, so a rebuild is required.

### 3.3 — `atlas-sync`

| Variable | Value |
|---|---|
| `BITCOIN_RPC_URL` | Depends on connectivity choice — see Part 4 |
| `BITCOIN_RPC_USER` | From Start9 Properties (step 0.1) |
| `BITCOIN_RPC_PASSWORD` | From Start9 Properties (step 0.1) |
| `TOR_SOCKS_PROXY` | Preset by the Blueprint to the in-container Tor (`socks5h://127.0.0.1:9050`) for Option B. Clear it for Option C |
| `CRYPTOCOMPARE_API_KEY` | Your free key (optional) |

## Part 4 — Connect the sync worker to your Start9 node

Pick **one** of the three options. **Option A is recommended** — it's the
fastest for the initial multi-day replay and keeps your RPC password inside
your own network.

### Option A (recommended) — run the worker at home, on your LAN

The worker is an ordinary Node process; it does not need to run on Render.
Run it on any always-on machine on the same network as the Start9 (a spare
laptop, a $150 mini-PC, a NAS that runs Node 20+).

1. **Suspend the Render worker:** `atlas-sync` → **Settings → Suspend**.
   (You keep the definition for later; you just won't run it there.)
2. Get the database's **external** connection string: `atlas-db` →
   **Info → External Database URL** (starts `postgres://…render.com…`).
3. On the home machine (Node 20+ and git installed):

   ```bash
   git clone https://github.com/YOUR_USERNAME/truenorth-atlas.git
   cd truenorth-atlas/server
   npm ci
   ```

4. Create `server/.env.local` (or export these in your shell):

   ```bash
   export DATABASE_URL='postgres://…external URL from step 2…'
   export PGSSLMODE=require
   export BITCOIN_RPC_URL='http://<start9-LAN-address>:8332'
   export BITCOIN_RPC_USER='<from Properties>'
   export BITCOIN_RPC_PASSWORD='<from Properties>'
   export CRYPTOCOMPARE_API_KEY='<optional>'
   ```

5. Start it:

   ```bash
   npm run worker
   ```

   You should see `schema ready`, then `price history ready`, then
   `initial sync progress` lines with climbing heights.

6. **Make it survive reboots** (Linux, systemd):

   ```ini
   # /etc/systemd/system/atlas-sync.service
   [Unit]
   Description=True North Atlas sync worker
   After=network-online.target

   [Service]
   WorkingDirectory=/home/YOU/truenorth-atlas/server
   EnvironmentFile=/home/YOU/truenorth-atlas/server/.env.local
   ExecStart=/usr/bin/node src/sync.js
   Restart=always
   RestartSec=15

   [Install]
   WantedBy=multi-user.target
   ```

   ```bash
   sudo systemctl daemon-reload
   sudo systemctl enable --now atlas-sync
   journalctl -u atlas-sync -f     # watch progress
   ```

### Option B — Tor, fully on Render (no home hardware)

The Blueprint's `atlas-sync` worker builds from `server/Dockerfile.worker`,
which bundles a Tor daemon inside the container — nothing to install. On
`atlas-sync`, set:

```bash
BITCOIN_RPC_URL='http://<your-node-onion-address>.onion:8332'
```

(the Tor Quick Connect address from step 0.1). `TOR_SOCKS_PROXY` is already
preset to `socks5h://127.0.0.1:9050` — the in-container Tor. The entrypoint
waits for Tor to bootstrap before starting the sync, so the first log lines
are Tor's, then `schema ready` etc.

The same pattern works on any home/VPS box: install Tor
(`sudo apt install tor`), set the two variables above. The `socks5h` scheme
matters — it resolves `.onion` names through Tor.

Expect the initial replay to run **several times slower** than LAN.

### Option C — VPN / tunnel

Put the Start9's LAN RPC behind Tailscale, WireGuard, or an SSH tunnel from
the worker host, then use the tunneled address as `BITCOIN_RPC_URL`. LAN-class
speed from anywhere in the tailnet.

## Part 5 — Watch the sync and verify the site

1. **Logs.** Wherever the worker runs, you'll see `initial sync progress`
   every ~5,000 blocks. Expect **several days to ~2 weeks** over LAN for the
   full replay (it pauses at every UTC day boundary to snapshot the UTXO set —
   that exactness is the product).
2. **The site is live immediately.** Open the `atlas-web` URL. Cards fill in
   as days finalize — early on you'll see 2010–2011 history appear, then it
   marches forward. The header shows the current synced block height.
3. **Sanity checks** (any terminal):

   ```bash
   API='https://atlas-api.onrender.com'
   curl $API/api/health            # → {"ok":true}
   curl $API/api/status            # → syncedHeight climbing, latestMetricsDay advancing
   curl $API/api/explorer/block/1  # → the first block, once synced
   ```

## Part 6 — Bootstrap admins and create your first API key

The `ADMIN_TOKEN` from step 3.1 is the **root credential** — the first admin
is established by controlling the deployment, not from inside the app.

1. Open the site → scroll to the footer → click **Admin** (or go to
   `<site-url>/#/admin` directly).
2. Paste the root `ADMIN_TOKEN` → **Unlock**. The header confirms you are
   signed in as **root**.
3. **Mint your personal admin token first:** **Admins** tab → enter your name
   → **Add admin** → copy the `tn_admin_…` token into your password manager.
   Do the same for anyone else who should manage keys — one token per person,
   individually revocable, with every API key attributed to its creator. Only
   root can add or revoke admins.
4. Lock the panel, then unlock again with your personal admin token for
   day-to-day work. Put root away.
5. **API Keys** tab → type a name for the consuming application (e.g.
   `sata-dashboard`) → **Create key**.
6. **Copy the key immediately** — it displays once; only its hash is stored.
7. Hand the key to your application. It authenticates every request:

   ```bash
   curl -H "X-API-Key: tn_live_…" $API/v1/address/bc1q…
   ```

8. The **API Reference** tab in the same panel shows every endpoint with a
   verified example response — send integrators there.
9. Revoke any key (or any admin, as root) instantly from the panel;
   revocation takes effect on the next request. Revoking an admin does NOT
   revoke the API keys they created — applications keep running; rotate those
   keys deliberately if needed.

---

### 6.5 — Enable email alerts (Resend)

1. In resend.com: **Domains → Add Domain** → `tnorth.com` → add the DKIM/SPF
   records it shows at your DNS provider → wait for **Verified**.
2. **API Keys → Create** → copy the `re_…` key.
3. On `atlas-api`, set `RESEND_API_KEY` and confirm `ALERTS_FROM_EMAIL`
   matches the verified domain (default `Atlas Alerts <alerts@tnorth.com>`),
   and `PUBLIC_API_URL` = the service's own URL. Redeploy.
4. Test: open any metric → "Get the signal" → set an alert with your own
   email → confirm from your inbox. The checker evaluates finalized days
   every 15 minutes and fires once per threshold crossing.
5. **Opening the doors:** alert and newsletter signups ship **disabled**. When
   you're ready, set `ALERT_SIGNUP_ENABLED=true` and/or
   `NEWSLETTER_SIGNUP_ENABLED=true` on `atlas-api` and redeploy — the forms
   appear on the site automatically and the endpoints start accepting.
   Flipping back off later blocks new signups only; existing subscribers,
   confirmed alerts, and unsubscribe links are never affected.
6. Every send (alerts, confirmations, newsletters — successes and failures)
   is recorded in the **Email Log** tab of the admin panel. Newsletters are
   drafted, chart-attached, tested, and scheduled from the **Newsletter** tab;
   recipients come from the site's footer subscribe form (double opt-in).

## Part 7 — Custom domain and embedding in tnorth.com

### 7.1 — Custom domain

1. `atlas-web` → **Settings → Custom Domains → Add** →
   `atlas.tnorth.com`.
2. At your DNS provider, add the CNAME record Render shows you. TLS certs are
   automatic.
3. If you also want a branded API host (optional): repeat on `atlas-api`
   with e.g. `api-atlas.tnorth.com`, then update `VITE_API_URL` and redeploy
   `atlas-web`.
4. Confirm `CORS_ORIGIN` on `atlas-api` lists every origin that will load
   the app.

### 7.2 — Embed as a module in tnorth.com

The app is hash-routed and iframe-friendly; `?embed=1` strips its own header
and footer so it inherits your page shell:

```html
<iframe
  src="https://atlas.tnorth.com/?embed=1"
  style="width:100%; height:100vh; border:0"
  title="True North Atlas">
</iframe>
```

Deep links work inside the embed too:
`…/?embed=1#/m/mvrv-z` (a specific metric), `…/?embed=1#/explorer` (the explorer).

## Part 8 — Post-sync housekeeping

Once `syncedHeight` reaches the chain tip:

- [ ] **Downsize the worker.** If you kept `atlas-sync` on Render, drop it
      from `standard` to `starter` — steady state is one block per ~10 minutes.
      If you ran it at home, it can stay there indefinitely.
- [ ] **Check database headroom.** `atlas-db` → **Metrics**. The UTXO set is
      ~190M rows; you provisioned 150 GB. If usage crosses ~80%, grow the disk
      (one click, no downtime).
- [ ] **Enable database backups** if not already on: `atlas-db` →
      **Recovery** — point-in-time recovery is worth it; a re-sync from
      genesis is the alternative.
- [ ] **Bookmark the admin panel** and store `ADMIN_TOKEN` in your password
      manager.

## Troubleshooting

| Symptom | Likely cause → fix |
|---|---|
| Worker logs `ECONNREFUSED` / timeouts to the node | Wrong `BITCOIN_RPC_URL`, node RPC disabled, or firewall. From the worker host: `curl -u user:pass -d '{"method":"getblockcount"}' http://<node>:8332/` should return a number |
| Worker logs `401 Unauthorized` from node | RPC user/password mismatch — recopy from Start9 Properties |
| `.onion` connections fail instantly | `TOR_SOCKS_PROXY` missing, Tor not running, or scheme is `socks5` instead of `socks5h` |
| Site loads but says "API unreachable" | `VITE_API_URL` wrong or set after the last build → fix it and **redeploy `atlas-web`** |
| Browser console shows CORS errors | Add the page's origin to `CORS_ORIGIN` on `atlas-api` (comma-separated, no spaces) |
| Explorer tx pages say node connection unavailable | The **API service** has no RPC credentials — that's optional but fill 3.1's RPC vars to enable it |
| Admin panel says "admin surface disabled" | `ADMIN_TOKEN` not set on `atlas-api` → set it and redeploy |
| `429 rate limited` on public endpoints | Working as designed (60/min per IP). Programmatic consumers should use an API key on `/v1` |
| Metrics look sparse / days missing | Normal during initial sync — days appear in order as the replay reaches them. Check `/api/status` for `latestMetricsDay` |
| Worker restarted mid-sync | Safe. All writes are transactional per block batch; it resumes exactly where it stopped, and reorg detection re-verifies the recent tip on every start |

## Reference — complete environment variable list

See `.env.example` in the repo for the documented master list. The ones you
will actually touch:

| Variable | Where | Purpose |
|---|---|---|
| `DATABASE_URL` | api, worker | Postgres connection (Blueprint wires it on Render; external URL for a home worker) |
| `BITCOIN_RPC_URL` / `_USER` / `_PASSWORD` | worker (api optional) | Your node |
| `TOR_SOCKS_PROXY` | worker (api optional) | Only for `.onion` nodes |
| `ADMIN_TOKEN` | api | Unlocks the admin panel / key management |
| `CORS_ORIGIN` | api | Origins allowed to call the API from a browser |
| `PUBLIC_RATE_LIMIT_PER_MIN` | api | Free-tier throttle (default 60; 0 disables) |
| `CRYPTOCOMPARE_API_KEY` | worker | Optional, better price-sync rate limits |
| `VITE_API_URL` | web (build-time) | Points the frontend at the API |
